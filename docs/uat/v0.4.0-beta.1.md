# v0.4.0-beta.1 user acceptance record

Date: 2026-07-21

The published release archives were tested as operator-facing artifacts rather
than from the repository checkout. Linux installation ran inside a systemd
cgroup limited to one CPU, 2 GB of memory, and 128 tasks; peak installer memory
was 301 MB. The gateway/browser pass remained bounded at one CPU, 1.5 GB, and
256 tasks; peak memory was 506 MB. macOS used a portable, checksum-verified
Node.js 24 runtime, a 1 GB Node heap, and single-worker package settings.

## Passed

- GitHub release archive and checksum verification.
- Linux and macOS archive installation.
- macOS upgrade, pointer rollback, and execution after rollback.
- Blank and scripted onboarding paths.
- Deterministic tool and multi-step plan execution.
- Run inspection, event-chain verification, signed audit verification, and
  audit-key rotation.
- Memory remember/search/recall/browse/open.
- Session create/message/rename/read/list.
- Goal create/update/list and improvement proposal/list.
- Security posture display and dangerous-change confirmation.
- State backup, mutation, restore, and post-restore validation.
- Public web fetch.
- Cold-start local Ollama verification with `qwen3-vl:4b`; the response
  completed in roughly 50 seconds under a 6 GB/1.5 CPU Ollama cap.
- Authenticated gateway status and project creation.
- Browser open/snapshot, explicit action approval, uncertain-outcome lockout,
  operator recovery, gateway restart, and stable-tab recovery.
- Proof, Sentinel, one-use capabilities, Rewind preview/apply, capsule
  export/verify/mock replay, counterfactual execution, and Darwin routing.

## Findings fixed after the release

- One-shot CLI browser commands kept the persistent Chromium context alive,
  preventing process exit and locking the profile against another CLI or
  gateway process.
- Restoring browser approval to the safer `true` value incorrectly required
  the same impact confirmation as disabling approval.
- The 20-second provider verification timeout was too short for a cold local
  4B Ollama model.
- Browser launch failures could be mislabeled as provider failures because
  Chromium diagnostic flags contain the word `Provider`.
- Installer output exposed an internal immutable directory identity as the
  release version, and source archives did not carry an exported commit ID.

## Remaining gaps

- The configured Ollama cloud model had been retired. The local provider path
  passed after the timeout fix, but cloud OAuth and API-key paths still need
  real-provider evidence.
- Windows installation remains untested because no Windows host was available.
- OAuth, API-key, and multi-day daily-use evidence remain outstanding.
- Localhost console clicking could not be driven by OpenClaw browser automation
  because that controller correctly blocks loopback/private navigation; the
  authenticated console APIs were exercised directly instead.
